## Grouped by failure category (per QA_RUNBOOK)

**Rules-engine / silent side effects (1):** swapping Back Squat → Step Ups
silently changed the Deadlift's load in the same session, undisclosed in the
confirmation message. Same category as the Program audit's 1.6 finding
(Bin action re-injecting/silently emptying unrelated days) — another
instance of a mutation changing more than what was requested without
telling the athlete. Per `SUPPORTED_ATHLETE_ACTIONS.md`'s pass criteria
("no silent side effects on other days/exercises/loads"), this is a genuine
FAIL, not a duplicate of a known gap.

**Accessibility (1):** the Remove-exercise flow's "apply to future weeks?"
scope buttons have no accessible label, unlike the same UI pattern in Swap
and Add. Inconsistent, not a total accessibility-tree blackout like the
Program audit's FRI finding, but still a real gap for anyone driving the
app non-visually.

**Data persistence / visible-plan-mismatch (1):** Session feedback submits
cleanly through the whole form but leaves no visible trace anywhere in the
app — the day card and the workout screen both look exactly as if nothing
was ever submitted. This directly contradicts pass criterion #4 ("the
visible plan changes to match") and Group 2.1's own test note ("Persists").
Whether the data is silently saved server-side and just not rendered back,
or genuinely dropped, is a diagnosis question — out of scope for this
tap-record-move-on pass.

**Not tested:** pass criterion #5 (survives app relaunch) — skipped per the
documented environment fact that plain relaunch after a seed session
white-screens; not rediscovering that known issue. "Play [exercise] demo"
buttons were seen but not tapped (video playback verification is outside
tap-record scope and risks getting stuck in a media player state mid-audit).
"Something hurts / no equipment" guided-change entry point was seen in the
Edit-exercises sheet but not exercised — functionally a re-route into the
same Swap flow already covered above under a different framing.
